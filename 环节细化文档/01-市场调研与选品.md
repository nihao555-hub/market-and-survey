# 环节①：市场调研 / 选品 — 构建方案

> 原则：最大化复用开源高分项目，自研部分压到最小。
> 自研只做 3 样小东西（见第四节），其余全是"拉库 + 配置 + 拼接"。
> 关联文档：《工具库-数据采集与爬虫.md》《工具库-选品数据源.md》

---

## 一、整体架构（数据流）

> 编排方案已定：**纯代码内编排（APScheduler + tenacity），不用 n8n、不上云**。详见《01-选品-项目拉取清单.md》。

```
        ┌──────── 纯代码编排（APScheduler定时 + tenacity重试，进程内）────────┐
        │                                                                      │
[采集层]│   [清洗层]          [存储层]        [分析层=我们的脑子]   [展示层]    │
Scrapling│  crawl4ai /        PostgreSQL      LLM(API) + 提示词 +   Superset/   │
+pydoll ─►  markitdown  ─────►(+定时快照)────► BSR估算表 + 利润公式 ─►Metabase  │
+curl_cffi│ 清成干净文本                        → 选品决策建议        或纯报告   │
pytrends │                                                                      │
        └──────────────────────────────────────────────────────────────────────┘
```

一句话：**采集→清洗→入库→分析→看板**，五层里有四层是直接拉开源，只有"分析层"是我们自己写，且主要是提示词+几十行公式。

## 二、要拉取的开源项目清单（pip install / git clone）

### A. 采集层（多引擎冗余，保证鲁棒性）
| 项目 | 安装方式 | 角色 | 许可证 |
|---|---|---|---|
| `D4Vinci/Scrapling` | `pip install scrapling` | **主引擎**：自适应解析，页面改版不易崩 | BSD-3 ✅ |
| `autoscrape-labs/pydoll` | `pip install pydoll` | 备用引擎：JS渲染+反检测 | MIT ✅ |
| `lexiforest/curl_cffi` | `pip install curl_cffi` | 快速通道：TLS指纹伪装，无渲染页 | MIT ✅ |
| `GeneralMills/pytrends` | `pip install pytrends` | Google Trends 趋势/季节性 | Apache ✅ |
| `FlareSolverr/FlareSolverr` | Docker(独立服务) | 验证码/Cloudflare 兜底 | MIT ✅ |
| `jhao104/proxy_pool` | Docker(独立服务) | 代理池/IP 轮换，反封禁 | MIT ✅ |

### B. 清洗层（抓完转成 LLM 能吃的干净文本）
| 项目 | 安装方式 | 角色 | 许可证 |
|---|---|---|---|
| `unclecode/crawl4ai` | `pip install crawl4ai` | 网页→LLM-ready markdown/结构化 | Apache ✅ |
| `microsoft/markitdown` | `pip install markitdown` | 文档/PDF→干净 markdown（备选） | MIT ✅ |

### C. 存储层（直接用，不自建）
| 项目 | 安装方式 | 角色 |
|---|---|---|
| PostgreSQL | Docker 一行 | 商品库 + 价格/BSR 历史快照 |
| （可选）Redis | Docker 一行 | 采集任务去重/缓存 |

### D. 编排层（纯代码，进程内，不上云）
| 项目 | 安装方式 | 角色 | 许可证 |
|---|---|---|---|
| `agronholm/apscheduler` | `pip install apscheduler` | 进程内定时任务，替代 n8n 的"定时触发" | MIT ✅ |
| `jd/tenacity` | `pip install tenacity` | 采集失败自动重试、多引擎降级 | Apache ✅ |
| （复杂时可选）`PrefectHQ/prefect` | `pip install prefect` | 需要 DAG/可视化时再上，可纯本地 | Apache ✅ |

### E. 展示层（不写前端报表）
| 项目 | 安装方式 | 角色 | 许可证 |
|---|---|---|---|
| `apache/superset` | Docker | 选品看板，许可证友好(优先) | Apache ✅ |
| `metabase/metabase` | Docker | 看板备选，拖拽易用 | AGPL ⚠️隔离/嵌入 |
| `pandas`+`plotly` | `pip install` | 最轻：代码内直接出图表/报告 | MIT/BSD ✅ |

> ⚠️ Metabase(AGPL) 以独立容器/嵌入方式部署，与闭源主体隔离，规避许可证传染。Superset(Apache) 更适合深度集成。

## 三、各层怎么拼（数据流转细节）

1. **APScheduler 定时任务**（如每小时）触发采集：
   - 抓 Amazon Best Sellers + Movers&Shakers（趋势爆品）
   - 抓目标类目商品详情页（BSR/价格/评分/评论数）
   - 抓评论正文（用于痛点分析）
2. **采集执行**：Scrapling 主抓，tenacity 控制失败重试并自动降级到 pydoll / curl_cffi（冗余）；全程走 proxy_pool 轮换 IP，被验证码挡则交 FlareSolverr 兜底。
3. **清洗**：crawl4ai 把页面转成干净结构化数据/markdown。
4. **入库**：写入 PostgreSQL；同时把当前 BSR/价格作为一条**快照**存入历史表（这样自己积累历史曲线，替代 Keepa）。
5. **趋势**：pytrends 拉品类 Google Trends 热度/季节性，入库。
6. **分析（我们的脑子）**：调用 LLM，按提示词产出：评论痛点/卖点、利润测算、蓝海/红海判断、选品建议。
7. **展示**：Superset/Metabase 连 PostgreSQL 出看板；或直接把"选品报告"推到飞书/邮件。

## 四、我们自己要造的轮子（只有这 3 样，都很小）

> 这是差异化与护城河所在，但工作量很小。

### 轮子1：亚马逊页面解析规则（小）
- 内容：定义"商品页/榜单页/评论页"要抽取的字段 + 选择器。
- 为什么小：Scrapling 的自适应能力会在页面改版后自动重定位元素，维护成本低。
- 量级：每类页面几十行配置。

### 轮子2：BSR → 销量估算表 + 利润公式（小）
- BSR估算：按类目建"BSR区间 → 月销量区间"映射表，用少量已知样本校准（初期可借 1 个付费工具的数值校准，之后甩开）。
- 利润公式：采购价+头程+平台佣金+FBA费+广告+退货率+汇率税费，纯算术，几十行。
- 量级：一张表 + 一个函数。

### 轮子3：LLM 分析提示词库（中，但只是写 prompt）
- 评论 → 痛点/卖点提炼
- 竞品数据 → 蓝海/红海判断 + 差异化机会点
- 综合 → 选品决策建议（是否上架/定价区间）
- 量级：几个精心设计的 prompt 模板，不是代码。

**除此之外，全部用开源拼装，不自研。**

## 五、为什么这样做自研最少（对照）

| 能力 | 传统做法(自研) | 我们的做法 |
|---|---|---|
| 爬虫框架/反爬 | 自己写一套 | Scrapling+pydoll 直接用 |
| 页面清洗 | 自己写解析 | crawl4ai/markitdown |
| 任务调度 | 自己写 cron+队列 | n8n 可视化配置 |
| 数据库 | - | PostgreSQL 直接用 |
| 报表前端 | 自己写 | Metabase 拖拽 |
| 趋势数据 | 自己对接 | pytrends |
| **业务洞察** | 自研 | **自研(提示词+小模型，唯一重点)** |

自研占比：约 5%~10%，其余靠拼装。
