# ① 选品环节 — 数据真实性与架构审计报告

> 用户四问严肃逐项回答，不粉饰。

## 一、用户输入什么 / 产出什么

### 用户输入
**最少**：一句自然语言
> "我想做蓝牙耳机这个品类，帮我做完整的选品调研。"

**Agent 反问澄清后还需 6 项**（这是我们当前 PoC 用 `auto_answer` 模拟的）：
1. 目标市场（US / EU / SEA / 全球）
2. 目标平台（Amazon / Walmart / Temu / Shopee / 独立站）
3. 资金预算（月度采购 + 单 SKU MOQ 阈值）
4. 物流方式（FBA / 海外仓 / 自发货）
5. 商家定位（白牌走量 / 自有品牌精品 / 代工贴牌）
6. 排除项（不做的品牌、专利雷区）

### Agent 产出
3 份文件，全部 UTF-8：

| 文件 | 内容 | 用途 |
|---|---|---|
| `agent_run_<TS>.log` | loguru 调试日志 | 排错 |
| `agent_transcript_<TS>.md` | 完整 transcript（Agent 思考 + 工具调用 + 结果） | 给运营/审计看过程 |
| `agent_final_<TS>.md` | 干净的最终决策报告 | 给商家看结论 |

**最终决策报告应该包含**（按方法论 8 阶段）：
- 候选品决策表（SKU + 售价 + 净利 + 毛利率 + 盈亏点 + 上架/观察/放弃）
- 主推 1-3 款产品定义（差异化点 + 参数表 + 定价节奏）
- 风险清单（专利/商标/政策/物流/退货）
- 90 天行动计划（D0-30 备货 / D31-60 测款 / D61-90 优化）
- 财务摘要（首批投入 / 月盈亏点 / 预期 ROI）

## 二、数据真实性审计（逐条标红）

我把 transcript 里出现的所有数字都做了 trace。

### ✅ 真实可信的数据（来自外部 API/抓取）

| 数据点 | 来源 | 真实性 |
|---|---|---|

| Amazon Top BSR 榜单 30+ 件 | botasaurus + 美国代理实抓 | ✅ **真**（B08JHCVHTY = Blink Plus Plan 真实存在） |
| Apple AirPods 4 (B0DGHMNQ5Z) 4.6★ / 29874 评 | Amazon 详情页抓 | ✅ **真** |
| Apple AirPods Pro 3 (B0FQFB8FMG) 4.5★ / 9912 评 | Amazon 详情页抓 | ✅ **真** |
| TOZO T6 (B07RGZ5NKS) 4.4★ / 225194 评 | Amazon 详情页抓 | ✅ **真** |
| 16+13 条 Apple/TOZO 真实评论文本 | 详情页底部公开评论区 | ✅ **真** |
| 价格中位 $25.99 / 评分中位 4.4 | analyze_market_structure 真算 | ✅ **真**（基于真实 33 件商品数据） |
| 痛点提炼："佩戴不适 5次/电池衰减 4次" | LLM 从真评论文本提炼 | ✅ **真**（但 count 是 LLM 估的，不是机器统计） |

### ⚠️ 半真半假（输入的"种子数"是 LLM 编的，公式是真算的）

| 数据点 | 问题 |
|---|---|
| **3 个候选品 SKU** "运动 ANC / 超舒适 ANC / 办公通话" | ⚠️ **完全是 LLM 凭印象想出来的产品方向**，不是从真实 BSR/搜索结果里映射的具体竞品 |
| **采购成本** $12 / $15 / $18 | ⚠️ **LLM 凭"售价 ×30~40%"经验拍的**，不是从 1688/Alibaba 真查的 |
| **monthly_sales_estimate** 1200 / 1500 / 1800 | ⚠️ **LLM 凭印象给的销量预估**，不是从 BSR→月销估算函数算出来的 |
| **MOQ 800** | ⚠️ **LLM 拍的**，不是从供应商 API 拿的 |
| **品牌候选名 AuralPace / FitBuds Pro** | ⚠️ **LLM 现编的品牌名**，没有真实的品牌可用性核查 |
| **品牌集中度 CR4=52%** | ⚠️ 算法基于"商品标题首词当品牌"，不是真实品牌字段 |
| **价格空白区分析** "$48-$85 仅 4 款" | 🟡 基于 33 件搜索结果分桶，**样本量太小**（应该 ≥100） |

### ❌ 完全凭空 / 编造的数据

| 数据点 | 问题 |
|---|---|
| **JBL Endurance Peak 4 ($99) "对标"** | ❌ **LLM 编的**，没真去查 JBL Endurance Peak 4 的真实价格和 ASIN |
| **"Soundcore P30i ($28) 对标办公通话"** | ❌ Soundcore P30i 真实价格是 $69.99（详见 transcript step 6 实抓数据），LLM 给的 $28 是错的 |
| **14 项成本里的常量**（关税 10%、ACOS 20%、退货 8%、汇率损失 1%、收款手续费 1.3% 等） | 🟡 **写在 `full_cost.py:DEFAULTS` 里的行业经验值**，不是 HTS 实查 / 真实店铺 ACOS / 真实退货统计 |

### 🔴 利润测算的整个链条是"LLM 输入 + 公式真算"

```
LLM 给输入参数 (sale_price=69.99, procurement_cost=15, MOQ=800, sales=1500)
   ↓
full_cost_breakdown() 用 DEFAULTS 里的固定常量真算
   ↓
输出 net_profit=16.2, margin=23.15%
```

**结果数字是公式真算的**（数学没错），**但输入参数是 LLM 想象的**。
所以最终利润是"基于假设的利润"，不是"基于真实数据的利润"。

## 三、过程是否高质量

### ✅ 高质量的部分
1. **真实抓取**：BSR、商品详情、评论、Trends 都是实拉的真数据
2. **方法论驱动**：Agent 第一步加载 `procurement-research.md`，按 8 阶段推进
3. **多引擎冗余**：scraper.py 9 级降级，遇到 503 自动切引擎
4. **模型切换**：前 11 步 Flash 编排，第 12+ 步切 Pro 做决策
5. **工具调用记录完整**：每次都有时间戳 + 参数 + 结果预览

### 🟡 中等质量的部分
1. **样本量偏小**：只抓了 ~40 件商品做"价格分布"，应该 ≥200
2. **评论数量**：累计 29 条，LLM 痛点 count 不是机器统计
3. **BSR 类目过宽**：用了"electronics"大类（含路由器、TV stick），不是耳机子类目
4. **Walmart/BestBuy 抓取**：Agent 调用了但因没用美国代理返回 0 条

### ❌ 低质量 / 凭空想象的部分
1. **候选品 SKU 是 LLM 拍的**，不是基于真实数据"差异化空隙"映射出的
2. **采购成本、MOQ、月销预估都是 LLM 经验值**，没有外部数据源
3. **品牌候选名是 LLM 编的**，没和 USPTO 商标库做"是否可注册"的真实查询
4. **专利检索返回空**（quick_ip_check 的 patents=[] 说明 Google Patents 抓取没解析出结果）

## 四、Agent 是否按 backend steering 运行

### Backend Steering 要求 vs 当前实现

| Steering 要求 | 当前实现 | 偏差 |
|---|---|---|
| **控制平面 / 数据平面分离**（HTTP 入队 + worker 跑 LLM） | ❌ 没有，单进程同步执行 | 严重偏差 |
| **任务队列**（BullMQ / Redis） | ❌ 无 | 严重偏差 |
| **pub/sub 流式事件**（GraphQL Subscription / Redis pub/sub） | ❌ 无，只有 print() | 严重偏差 |
| **streamText + UIMessageChunk** | ❌ 用 `stream=False`，整段返回 | 严重偏差 |
| **chunk 序列号 + catch-up 缓存** | ❌ 无 | 严重偏差 |
| **可取消**（abortController + cancel pub/sub） | ❌ 无 | 严重偏差 |
| **多模型 RegistryService**（catalog + custom） | 🟡 只用了 DeepSeek 一家 | 部分对齐 |
| **DEFAULT_FAST / DEFAULT_SMART** 模型切换 | ✅ 实现了（Flash/Pro 按 step 切） | 对齐 |
| **stopWhen + step 上限** | ✅ 实现了（MAX_STEPS=28） | 对齐 |
| **逐 step 扣费** | ❌ 无计费逻辑 | 严重偏差 |
| **prompt cache breakpoint** | ❌ 无 | 偏差 |
| **元工具 learn_tools / execute_tool**（大规模工具） | ➖ 当前 14 个工具，平铺 OK | 不需要 |
| **Skill 系统**（markdown + load_skills） | ✅ 实现了（procurement-research.md） | 对齐 |
| **持久化 user/assistant message + parts JSONB** | ❌ 内存里 list，没入库 | 严重偏差 |
| **Thread + activeStreamId + 队列限流** | ❌ 无多会话概念 | 严重偏差 |
| **失败重试** | ✅ tenacity 装饰了 scraper | 对齐 |

### 总体评价

**当前 Agent 是"PoC 同步脚本"，不是 backend steering 描述的"商业级 Agent 系统"**。

具体说：
- ✅ **方法论 + 工具循环 + 模型切换**这三块对齐了 steering
- ❌ **基础设施**（队列/pubsub/流式/取消/计费/持久化）**完全没做**

这是**第一阶段 PoC 验证业务流程的合理选择**——先把"工具能跑通、数据能采集、决策能产出"证明出来，再上基础设施。

但要做成 SaaS 上线，还需要补：
1. FastAPI/GraphQL 接口层（控制平面）
2. Celery / Dramatiq / arq 任务队列（数据平面）
3. Postgres 持久化 messages/threads
4. Redis pub/sub 推 stream-chunk 给前端
5. SSE 端点 + 前端 readUIMessageStream
6. 计费/限流/取消机制

## 五、当务之急的修复建议（按优先级）

### P0（解决"凭空数字"）
1. **接 1688 / Alibaba 采购成本 API**（或人工录入供应商报价库）
   → 候选品的 procurement_cost 不再是 LLM 拍脑袋
2. **用真实的 BSR→月销估算映射表替换简易公式**
   → monthly_sales_estimate 用 `estimate_monthly_sales_from_bsr` 真算
3. **强制 LLM 候选品必须从真实抓到的商品里选**（不许凭空创造）
   → 修改 system prompt：候选 SKU 必须引用具体 ASIN

### P1（提升数据质量）
4. **抓耳机子类目 BSR**（不是 electronics 大类）
   → 加入 `https://www.amazon.com/Best-Sellers-Electronics/zgbs/electronics/172541/` 等子类目
5. **样本量提到 100+**（多翻几页 / 多个关键词聚合）
6. **评论拓展到 50-100 条**（5-10 个 ASIN × 8-13 条 = 40-130）

### P2（架构升级到 backend steering）
7. FastAPI + Celery 把同步循环改成异步流式
8. Postgres 持久化 + Redis pub/sub
9. 前端 SSE 接收 stream-chunk

### P3（数据校准）
10. **DEFAULTS 里的行业常量**接真实 API（HTS 关税表、店铺真实 ACOS、真实退货率）
